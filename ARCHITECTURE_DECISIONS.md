# 架构与产品决策

## ADR-001 温控能力与实体设备解耦

当前模型只有风扇，没有空调。系统使用统一的温控策略计算 `on/off/hold`，再由 `FanClimateAdapter` 执行动作。阶段 2 接入空调时新增 HVAC adapter，不修改阈值、回差、GUI或历史数据逻辑。

采用目标温度加回差：温度达到 `目标 + 回差` 时启动，降至 `目标 - 回差` 时停止，中间区间保持状态，避免临界点频繁启停。

这一设计参考 Home Assistant 将设备能力抽象为 climate entity 的思路，但代码为本项目独立实现：<https://github.com/home-assistant/core>

## ADR-002 门禁失败采用时间窗聚合

同一 `subject_key` 在默认10分钟内失败3次后进入 `suspicious_people`，管理员处理后状态改为 `resolved`。失败记录、聚合告警和普通事件分表保存，便于设置不同保留期。

参考 Frigate 将检测记录、审查事件和保留策略分离的产品思路：<https://github.com/blakeblackshear/frigate>

## ADR-003 图片与结构化数据分离

SQLite保存人员、设备值、识别标签、可信度、时间和图片路径，不直接保存大图片二进制。未来图片写入本地目录、NAS或对象存储，数据库只保存路径、哈希和元数据。保留期清理接口负责删除过期结构化记录，阶段 2 再增加对应文件清理器。

## ADR-004 人脸样本的规模化路线

阶段 1 使用 `face_key` 验证完整门禁业务。真实人脸模块应将多张注册照片转换为归一化特征向量，按余弦相似度匹配；不要逐张加载原图比较。模型边界参考 InsightFace：<https://github.com/deepinsight/insightface>

小规模（几十人）可在内存中向量化批量计算；规模增长后迁移PostgreSQL + pgvector，并使用HNSW近似索引：<https://github.com/pgvector/pgvector>

## 复杂度审计

| 操作 | 当前复杂度 | 说明 |
| --- | --- | --- |
| 按ID修改人员 | `O(log n)` | SQLite主键索引 |
| 查询授权状态 | `O(log n)` | `authorized`索引 |
| 统计某身份近期失败 | `O(log n + k)` | `subject_key + occurred_at`复合索引；`k`为窗口内记录 |
| 获取最近事件/遥测 | `O(log n + k)` | 类别/指标与时间复合索引 |
| 写入一组遥测 | `O(m)` | 单事务批量写入；`m`为设备指标数 |
| 阶段1人脸匹配 | `O(n)` | 适合小组演示规模 |
| 阶段2 HNSW检索 | 近似 `O(log n)` | 以部分召回率换取大规模查询速度 |

首页不再为每个指标单独建立数据库连接；四项遥测使用一次批量事务，并在同一请求中只使用一个设备快照。

## 商业价值边界

当前产品的可展示价值是“本地优先、可离线、隐私友好、可接旧设备”的小型空间管理平台，适用于宿舍、实验室、小办公室和商铺原型。商业化前仍需补充管理员登录、权限分级、加密、备份、隐私告知、摄像头合规、多场所管理和远程升级。
